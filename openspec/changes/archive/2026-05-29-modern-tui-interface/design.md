## Context

NanoHermes 当前使用简单的 `input()` + `print()` 进行终端交互，缺乏现代 TUI 应有的用户体验。项目已集成 `prompt_toolkit` 和 `rich`，但未充分利用其能力。Hermes Agent 参考实现展示了基于 React + Ink 的高级 TUI，但我们需要 Python 原生方案。

**当前状态**:
- `src/cli/tui_chat.py`: 基础流式输出，简单工具摘要
- 依赖: `prompt_toolkit`（已安装），`rich`（已安装）
- 缺失: 动态布局、加载动画、进度指示器、智能补全、语法高亮

**约束**:
- Python 3.11+ 原生实现
- 不能使用 React/TypeScript（需要 Python 方案）
- 必须与现有会话、工具、记忆系统兼容

## Goals / Non-Goals

**Goals:**
- 构建现代化 TUI，支持流式 Markdown 渲染、动态工具可视化、智能输入补全
- 实现响应式布局，自适应终端尺寸
- 提供加载动画和进度指示器
- 保持与现有系统的完全兼容
- 编写完整的测试用例（单元测试 + 集成测试）

**Non-Goals:**
- 不实现完整的 React-like 组件系统（Python 限制）
- 不支持鼠标交互（专注于键盘驱动）
- 不实现多窗口/标签页功能
- 不替换现有的会话存储和工具执行逻辑

## Decisions

### 1. 架构：三层 TUI 设计

**决策**: 采用分层架构
- **Core Layer**: `prompt_toolkit` 处理输入、补全、历史记录
- **Rendering Layer**: `rich` + 自定义 ANSI 转义码处理输出
- **Component Layer**: 自定义 Ink UI 风格组件（Python 实现）

**理由**: 
- `prompt_toolkit` 是 Python 中最成熟的终端 UI 库
- `rich` 提供优秀的 Markdown 渲染和表格功能
- 自定义组件层提供灵活性，避免过度依赖

**替代方案**:
- ❌ 纯 ANSI 转义码：太底层，维护成本高
- ❌ 仅用 `rich`: 缺乏输入处理能力
- ❌ Textual: 学习曲线陡峭，过度复杂

### 2. 流式输出：打字机效果 + 增量渲染

**决策**: 实现流式 Markdown 渲染器
- 使用 `rich.markdown` 进行增量解析
- 打字机效果：逐字符输出，可配置速度
- 支持代码块语法高亮（通过 `pygments`）

**理由**: 用户期望实时反馈，而非等待完整响应

### 3. 输入补全：上下文感知

**决策**: 实现多层补全系统
- 命令补全：`/help`, `/clear`, `/resume` 等
- 文件路径补全：`./`, `../`, 自动补全
- 上下文感知：根据当前对话状态提供建议

**理由**: 提升用户效率，减少输入错误

### 4. 工具可视化：实时状态面板

**决策**: 实现动态工具调用显示
- 工具开始：显示加载动画
- 工具执行：显示进度/状态
- 工具完成：显示结果摘要

**理由**: 用户需要了解 Agent 在做什么

### 5. 响应式布局：终端尺寸自适应

**决策**: 监听终端尺寸变化，动态调整布局
- 最小宽度：80 字符
- 动态面板宽度：根据终端宽度调整
- 窗口调整事件处理

**理由**: 支持不同终端尺寸和分屏场景

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| `prompt_toolkit` 与现有代码冲突 | 隔离 TUI 模块，使用适配器模式 |
| 流式输出性能问题 | 使用缓冲区和异步渲染 |
| ANSI 转义码跨平台兼容性 | 使用 `rich` 抽象层，测试 Windows/Linux/macOS |
| 测试复杂性 | 使用 `pytest` + `prompt_toolkit` 测试工具，模拟终端 |
| 学习曲线 | 编写详细文档和示例 |

## Migration Plan

1. **阶段 1**: 实现核心 TUI 组件（保持旧 CLI 可用）
2. **阶段 2**: 集成流式输出和工具可视化
3. **阶段 3**: 添加输入补全和响应式布局
4. **阶段 4**: 编写测试，默认启用新 TUI
5. **回滚**: 保留 `--cli` 标志，可随时回退到旧模式
