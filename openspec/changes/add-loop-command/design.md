## Context

NanoHermes 当前的对话循环（`ConversationLoop`）是单次触发的：用户发送消息 → AI 调用模型 → 执行工具调用链 → 返回最终响应 → 等待用户下一次输入。这种模式适合一次性任务，但对于需要持续监控、轮询或迭代改进的场景非常不便。

现有基础设施：
- `ConversationLoop`（`src/conversation/loop.py`）：核心对话循环，支持最大迭代次数（90次）
- `cronjob` 工具（`src/tools/impls/cronjob_tool.py`）：定时任务管理，支持多种调度格式（间隔、cron、ISO 时间戳）
- `EventBus`（`src/conversation/events.py`）：事件驱动解耦，支持拦截器模式
- 会话存储（`src/session/`）：SQLite + JSONL 双存储，支持会话恢复

Claude Code 的 `/loop` 命令提供了三种模式：
1. 固定间隔 + 用户提示：按 cron 表达式定期执行
2. 动态间隔 + 用户提示：AI 根据上下文自动选择等待时间
3. 内置维护提示：自动处理未完成工作、PR 检查、代码清理

## Goals / Non-Goals

**Goals:**
- 提供 `/loop` 命令，支持固定间隔和动态间隔两种调度模式
- 支持用户自定义提示和内置维护提示
- 支持 `loop.md` 文件自定义默认提示
- 循环作用域限定在当前会话，支持 `--resume` 恢复
- 提供 `/stop-loop` 命令停止循环
- 复用现有 `cronjob` 工具的调度能力
- 循环执行时显示进度和状态

**Non-Goals:**
- 不实现云端调度（那是 Routines 的职责）
- 不实现跨会话持久化（超出 7 天的循环自动过期）
- 不修改 `ConversationLoop` 的核心逻辑
- 不实现复杂的权限控制（继承当前会话的权限）
- 不实现并行循环（同一时间只允许一个活跃循环）

## Decisions

### 1. 新增 `src/loop/` 模块而非扩展现有模块

新增独立模块包含：
- `manager.py` — 循环生命周期管理
- `prompt.py` — 内置维护提示和 `loop.md` 加载
- `interval_parser.py` — 间隔表达式解析

**理由：**
- 循环管理是独立的关注点，不属于对话循环、TUI 或工具系统
- 保持模块职责单一，便于测试和维护
- 与 Claude Code 的架构一致（`/loop` 是独立功能）

**替代方案：**
- 扩展到 `cronjob_tool.py`：会导致工具文件过大，且循环不是纯粹的定时任务
- 扩展到 `tui.py`：会使 TUI 文件超过 300 行限制，违反编码规范

### 2. 间隔解析支持多种格式

支持以下间隔格式：
- 持续时间：`30s`, `5m`, `2h`, `1d`
- 自然语言：`every 5 minutes`, `every 2 hours`
- Cron 表达式：`*/5 * * * *`（5 字段）

解析后统一转换为秒数用于调度。

**理由：**
- 与现有 `cronjob` 工具的 `_parse_schedule()` 保持一致
- 用户可以使用最方便的格式
- 秒级精度足够，cron 的最小粒度是分钟

**间隔验证规则：**
- 最小间隔：60 秒（1 分钟），防止 API 滥用
- 最大间隔：86400 秒（24 小时），防止忘记停止
- 秒单位自动向上取整到分钟（cron 限制）

### 3. 动态间隔由 AI 自行决定

当用户不提供间隔时，循环进入"自 paced"模式：
- AI 在每次迭代结束时自行决定下一次等待时间
- 等待范围：1 分钟到 1 小时
- AI 在响应中说明选择的间隔和理由

**实现方式：**
- 在系统提示中添加循环模式说明
- AI 在响应末尾输出 `__next_interval: 5m__` 标记
- `LoopManager` 解析该标记并设置下一次调度

**理由：**
- 适合不确定完成时间的场景（等待 CI、等待 PR 审查）
- AI 可以根据当前状态智能调整等待时间
- 无需额外的模型调用或复杂逻辑

**替代方案：**
- 固定默认间隔（如 10 分钟）：不够灵活
- 使用单独的分类模型决定间隔：增加复杂度

### 4. 内置维护提示设计

默认维护提示按优先级处理：
1. 继续会话中未完成的工作
2. 检查当前分支的 PR 状态（CI 状态、审查评论、合并冲突）
3. 无待办事项时执行清理（bug 修复、代码简化）

**安全约束：**
- 不启动新的主动工作（只处理已有上下文中的任务）
- 不可逆操作（push、delete）需要用户已授权
- 递归调度禁止（循环执行的会话不应再创建新循环）

**理由：**
- 与 Claude Code 的维护提示行为一致
- 确保循环不会意外启动用户未授权的工作
- 防止无限递归调度

### 5. `loop.md` 文件支持

查找顺序：
1. `.claude/loop.md` — 项目级，优先
2. `~/.nanohermes/loop.md` — 用户级

**理由：**
- 与 `CLAUDE.md` 的文件位置约定一致
- 项目级优先允许团队共享循环策略
- 用户级提供个人默认设置

**文件格式：**
- 纯 Markdown，无特殊结构要求
- 最大 25,000 字节（超出截断）
- 修改后下次迭代生效

### 6. 循环执行机制

循环执行复用现有的 `_run_conversation_loop()` 方法：

```
用户输入 /loop 5m "检查部署状态"
    │
    ▼
LoopManager.create_loop(interval="5m", prompt="检查部署状态")
    │
    ▼
保存到会话存储（JSONL 元数据）
    │
    ▼
TUI 显示循环状态指示器
    │
    ▼
等待间隔到期
    │
    ▼
_run_conversation_loop(prompt)  ← 复用现有方法
    │
    ▼
显示结果，等待下一次间隔
    │
    ▼
重复直到用户停止或过期
```

**设计理由：**
- 复用现有对话循环，不重复实现
- 循环提示作为用户消息注入，保持上下文连续
- 循环元数据存储在会话中，支持恢复

### 7. 循环状态存储

循环信息存储在会话 JSONL 文件中作为系统消息：

```json
{
  "role": "system",
  "content": "__loop_meta__",
  "meta": {
    "type": "loop",
    "loop_id": "abc12345",
    "interval": "5m",
    "prompt": "检查部署状态",
    "created_at": "2026-07-06T23:00:00Z",
    "expires_at": "2026-07-13T23:00:00Z",
    "status": "active"
  }
}
```

**理由：**
- JSONL 是会话的事实存储，无需额外存储
- 恢复会话时可重建循环状态
- 7 天过期自动清理

### 8. 停止机制

两种停止方式：
1. `/stop-loop` 命令：显式停止
2. `Esc` 键：在等待下一次迭代时清除待调度的循环

**实现方式：**
- `LoopManager.stop_loop()` 设置状态为 `stopped`
- TUI 主循环检查循环状态，跳过已停止的循环
- 等待中的循环通过 `Esc` 清除待触发的调度

### 9. 与 cronjob 工具的关系

`/loop` 和 `cronjob` 工具是互补关系：

| 特性 | `/loop` | `cronjob` 工具 |
|------|---------|---------------|
| 作用域 | 当前会话 | 全局（跨会话） |
| 执行上下文 | 当前会话上下文 | 新会话（无上下文） |
| 持久化 | 会话级（7 天） | 文件级（永久） |
| 提示格式 | 自然语言 | 自包含提示 |
| 使用场景 | 快速轮询、监控 | 长期自动化任务 |

**集成方式：**
- `LoopManager` 内部复用 `cronjob` 的间隔解析逻辑
- 不直接调用 `cronjob` 工具函数，避免工具调用链
- 循环元数据格式与 cronjob 任务格式不同，各自独立

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| 循环执行时用户无法输入新消息 | 循环在用户"回合之间"执行，不阻塞用户输入 |
| 忘记停止循环导致 API 滥用 | 7 天自动过期，最小间隔 1 分钟 |
| 循环提示与当前上下文冲突 | 循环提示作为用户消息注入，AI 自行判断上下文相关性 |
| 循环执行失败导致中断 | 单次失败不停止循环，记录错误继续 |
| 恢复会话时循环状态丢失 | 循环元数据存储在 JSONL 中，随会话持久化 |
| 多个循环同时运行 | 同一时间只允许一个活跃循环，新循环需先停止旧循环 |
| `loop.md` 文件包含恶意内容 | 与 `CLAUDE.md` 同等信任级别，用户自行控制 |
