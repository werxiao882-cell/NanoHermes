## 新增需求

### 需求：定期审查对话模式

系统应定期分析对话模式，识别创建或更新技能的机会。

#### 场景：对话达到阈值触发技能审查
- **当** 对话达到 10 轮或距离上次审查已过 30 分钟
- **则** 系统在后台线程中触发技能审查
- **则** 系统分析对话中的重复工作流和最佳实践

#### 场景：技能审查识别新技能机会
- **当** 对话包含可复用的工作流（如多步骤调试过程）
- **则** 系统生成技能提议，包含名称、描述和内容
- **则** 系统在 INFO 级别记录提议，供用户查看
- **则** 系统不自动创建技能（需要用户批准）

#### 场景：技能审查识别更新机会
- **当** 对话改进了现有技能的方法
- **则** 系统生成更新提议，包含差异对比
- **则** 系统在 INFO 级别记录提议

### 需求：技能审查使用受限的工具访问

系统应限制 fork 代理的工具访问，仅允许技能相关工具。

#### 场景：工具白名单强制执行
- **当** fork 代理尝试调用工具
- **则** 系统仅允许：`memory`、`skill_manage`、`skill_view`、`skills_list`
- **则** 系统拒绝所有其他工具调用，返回错误消息

#### 场景：代理遵守工具限制
- **当** fork 代理需要使用受限工具（如 `terminal`）
- **则** 代理使用允许的工具绕过限制
- **则** 代理不尝试绕过限制

### 需求：技能审查限制迭代次数

系统应将 fork 代理限制为最多 5 次迭代，防止无限循环。

#### 场景：代理在迭代限制内完成
- **当** fork 代理在 3 次迭代内完成分析
- **则** 系统返回最终结果
- **则** 系统在 DEBUG 级别记录完成

#### 场景：代理达到迭代限制
- **当** fork 代理达到 5 次迭代仍未完成
- **则** 系统终止代理执行
- **则** 系统记录关于迭代限制的警告
- **则** 系统返回部分结果（如果有）

### 需求：技能审查集成 Curator

系统应实例化并定期调用 `Curator` 管理技能生命周期。

#### 场景：TUI 启动时初始化 Curator
- **当** TUI 应用启动
- **则** 系统使用配置的参数创建 `Curator` 实例
- **则** 系统调用 `maybe_run()` 检查是否需要审查

#### 场景：Curator 定期执行
- **当** 距离上次 Curator 运行已过 7 天
- **则** 系统调用 `maybe_run()`，触发 `_run_review()`
- **则** Curator 评估技能使用情况并转换状态（active → stale → archived）

#### 场景：Curator 遵守空闲阈值
- **当** 距离上次运行不足 24 小时
- **则** `maybe_run()` 返回 False，不执行审查
- **则** 系统避免不必要的开销

### 需求：技能审查可配置

系统应允许用户通过配置控制技能审查行为。

#### 场景：技能审查被禁用
- **当** `background_tasks.skill_review.enabled` 设置为 `false`
- **则** 系统不触发技能审查
- **则** Curator 不被实例化

#### 场景：自定义审查间隔
- **当** `background_tasks.skill_review.interval_minutes` 设置为 60
- **则** 系统每 60 分钟触发一次审查，而非默认的 30 分钟
