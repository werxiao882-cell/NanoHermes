## 新增需求

### 需求：会话结束时自动提取记忆

系统应在会话结束时自动审查最近的对话消息，提取值得长期保存的信息。

#### 场景：会话结束且有值得记忆的内容
- **当** 用户退出对话（通过 `/quit`、Ctrl+D 或会话超时）
- **则** 系统审查最近 10 轮对话（20 条消息）
- **则** 系统提取用户偏好、重要事实和项目上下文
- **则** 系统通过 `memory` 工具将提取的信息写入 MEMORY.md 或 USER.md

#### 场景：会话结束但没有值得记忆的内容
- **当** 用户退出简短或琐碎的对话
- **则** 系统判断没有值得保存的信息
- **则** 系统不修改记忆文件

#### 场景：记忆提取遵循去重规则
- **当** 提取的信息已存在于记忆中
- **则** 系统不创建重复条目
- **则** 如果新信息更具体，系统可以更新现有条目

### 需求：记忆提取使用截断的上下文

系统应限制发送给 LLM 的上下文，控制成本并避免上下文溢出。

#### 场景：长消息被截断
- **当** 对话消息超过 500 字符
- **则** 系统在发送给 LLM 前将消息截断到 500 字符
- **则** 系统追加 "..." 表示已截断

#### 场景：强制执行 token 限制
- **当** 进行记忆提取 LLM 调用
- **则** 系统设置 `max_tokens=1000` 限制响应长度
- **则** 系统优雅处理截断的响应

### 需求：记忆提取在后台线程运行

系统应在独立线程中执行记忆提取，避免阻塞主对话循环。

#### 场景：后台线程执行
- **当** 会话结束且触发记忆提取
- **则** 系统生成守护线程执行提取
- **则** 主线程立即返回，不等待
- **则** 提取异步完成

#### 场景：后台线程失败被隔离
- **当** 记忆提取线程遇到错误
- **则** 系统在 WARNING 级别记录错误
- **则** 系统不崩溃或影响其他操作
- **则** 主应用继续正常运行

### 需求：记忆提取可配置

系统应允许用户通过配置启用/禁用自动记忆提取。

#### 场景：记忆提取被禁用
- **当** `background_tasks.memory_flush.enabled` 设置为 `false`
- **则** 系统不在会话结束时触发记忆提取
- **则** 用户必须手动使用 `memory` 工具保存信息

#### 场景：记忆提取被启用（默认）
- **当** `background_tasks.memory_flush.enabled` 未设置或设置为 `true`
- **则** 系统在会话结束时触发记忆提取
