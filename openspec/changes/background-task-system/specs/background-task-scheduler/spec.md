## 新增需求

### 需求：统一的后台任务调度器

系统应提供集中式的 `BackgroundTaskScheduler`，管理所有后台任务。

#### 场景：调度器初始化
- **当** TUI 应用启动时
- **则** 系统创建 `BackgroundTaskScheduler` 实例
- **则** 调度器使用配置的 `max_concurrent`（默认：2）初始化
- **则** 调度器创建内部信号量用于并发控制

#### 场景：任务注册
- **当** 后台任务模块加载时
- **则** 任务通过 `register_task(name, handler, trigger)` 注册到调度器
- **则** 调度器存储任务元数据（名称、处理函数、触发条件）

#### 场景：调度器关闭
- **当** TUI 应用退出时
- **则** 调度器等待运行中的任务完成（超时：10s）
- **则** 调度器终止剩余任务
- **则** 调度器释放资源

### 需求：并发控制

系统应限制同时运行的后台任务数量，防止资源耗尽。

#### 场景：并发限制生效
- **当** 已有 2 个后台任务运行（max_concurrent=2）
- **则** 第三个任务请求被排队
- **则** 第三个任务在运行中的任务完成后启动
- **则** 调度器在 DEBUG 级别记录队列状态

#### 场景：任务完成释放槽位
- **当** 运行中的后台任务完成
- **则** 调度器释放信号量槽位
- **则** 调度器启动下一个排队任务（如果有）

#### 场景：任务超时
- **当** 后台任务运行超过配置的超时时间（默认：300s）
- **则** 调度器终止任务线程
- **则** 调度器记录超时警告
- **则** 调度器释放信号量槽位

### 需求：事件驱动的任务触发

系统应根据对话事件触发后台任务。

#### 场景：LOOP_END 事件触发记忆刷写
- **当** `ConversationLoop` 发出 `LOOP_END` 事件
- **则** 调度器检查记忆刷写是否启用
- **则** 调度器检查会话是否有足够消息（>= 10）
- **则** 如果条件满足，调度器触发记忆刷写任务

#### 场景：LOOP_END 事件触发技能审查
- **当** `ConversationLoop` 发出 `LOOP_END` 事件
- **则** 调度器检查技能审查是否启用
- **则** 调度器检查审查间隔是否已到（10 轮或 30 分钟）
- **则** 如果条件满足，调度器触发技能审查任务

#### 场景：多个任务同时触发
- **当** LOOP_END 同时触发记忆刷写和技能审查
- **则** 调度器将两个任务排队
- **则** 调度器并发运行任务（最多 max_concurrent 个）
- **则** 调度器在任务启动之间添加 1 秒延迟，避免 API 速率限制

### 需求：后台线程执行

系统应在独立的守护线程中执行所有后台任务。

#### 场景：任务在守护线程中运行
- **当** 调度器启动后台任务
- **则** 系统创建 `threading.Thread`，设置 `daemon=True`
- **则** 线程运行任务处理函数
- **则** 主线程继续执行，不阻塞

#### 场景：任务异常处理
- **当** 后台任务抛出异常
- **则** 线程捕获异常
- **则** 系统在 ERROR 级别记录异常和堆栈跟踪
- **则** 线程优雅终止
- **则** 调度器释放信号量槽位

#### 场景：任务日志
- **当** 后台任务启动
- **则** 系统在 INFO 级别记录："后台任务启动：<task_name>"
- **当** 后台任务完成
- **则** 系统在 INFO 级别记录："后台任务完成：<task_name>（耗时：Xs）"

### 需求：任务配置

系统应允许细粒度配置每个后台任务。

#### 场景：全局启用/禁用
- **当** `background_tasks.enabled` 设置为 `false`
- **则** 调度器不触发任何任务
- **则** 所有后台任务功能被禁用

#### 场景：单个任务启用/禁用
- **当** `background_tasks.memory_flush.enabled` 设置为 `false`
- **则** 调度器不触发记忆刷写任务
- **则** 其他任务（技能审查等）如果启用则仍然运行

#### 场景：自定义并发限制
- **当** `background_tasks.max_concurrent` 设置为 3
- **则** 调度器允许最多 3 个任务同时运行
- **则** 调度器使用 max_concurrent=3 的信号量

#### 场景：自定义超时
- **当** `background_tasks.task_timeout_seconds` 设置为 600
- **则** 调度器在 600 秒后终止任务
- **则** 调度器记录超时警告

### 需求：任务状态监控

系统应提供后台任务执行状态的可见性。

#### 场景：查询运行中的任务
- **当** 用户或系统查询调度器状态
- **则** 调度器返回当前运行中的任务列表
- **则** 调度器包含任务名称、启动时间和运行时长

#### 场景：查询任务历史
- **当** 用户或系统查询任务历史
- **则** 调度器返回最近 10 个完成的任务
- **则** 调度器包含任务名称、运行时长和成功/失败状态

#### 场景：TUI 状态命令集成
- **当** 用户运行 `/status` 命令
- **则** 系统显示运行中的后台任务数量
- **则** 系统显示最近完成的任务（如果有）
