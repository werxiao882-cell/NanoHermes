## 🎯 项目概览

**JoyAI-VL-Interaction** 是京东开源的 **8B 规模实时视频-语言交互系统**（Apache 2.0 协议）。

它是业界首个完整开源的**视觉交互模型**——不仅公开了模型权重，还公开了**训练方法、时间对齐数据集**和**完整的可部署系统**。

---

## ❌ 核心要解决的问题

> 现实世界的关键时刻不会等你提问。

当前 AI 大多是**回合制**（turn-based）设计——只有被问才回答，即使看似交互的视频通话应用，本质上也是问答系统。但很多场景下，用户无法主动发问：

- 监控画面中突然起火
- 视频通话中一闪而过的关键表情
- 直播中观众想要的商品瞬间出现

**JoyAI 提出新范式：让 AI "像人一样在场"**
> 持续观看画面，自己判断什么时候该说话、什么时候保持安静，遇到难题时交给更强的模型处理。

**五个可插拔服务：**

## ⏱️ 实时推理是如何实现的？

### 帧率与延迟指标

| 指标 | 数值 |
|------|------|
| **视频输入帧率** | **~1 FPS**（每秒约 1 帧 JPEG 图像） |
| **决策频率** | **每秒 1 次**（speak / silent / delegate） |
| **响应延迟** | **< 1 秒**（从画面变化到模型开口） |

### 每一秒是怎么"看"的？

系统采用**时间门控**策略，每 1 秒尝试向模型发送一帧：

```python
# video_processor.py 核心逻辑
async def recv(self):
    frame = await self.track.recv()  # 从 WebRTC 获取帧
    
    # 时间门控：只有间隔 ≥ 1 秒才触发推理
    interval_sec = 1.0
    time_since_last = now - self._last_process_time
    
    if time_since_last >= interval_sec:
        # 异步发送给 VLM，不阻塞视频流接收
        asyncio.create_task(
            self.vlm_service.process_frame(pil_img, ...)
        )
        self._last_process_time = now  # 重置计时器
```

关键点：**不是"等推理完再看下一帧"**，而是**固定节奏尝试发送**，推理在后台异步进行。

### 🚫 如果推理不及时怎么办？（核心保护机制）

系统实现了**三层防积压策略**：

#### 1. 忙时跳过（Busy Skip）

```python
# vlm_service.py 核心逻辑
async def process_frame(self, image, ...):
    # 如果上一轮推理还在进行中，直接跳过！
    if self._processing_lock.locked():
        logger.debug("VLM busy, skipping frame")
        return  # 不排队、不等待、直接丢弃
    
    async with self._processing_lock:
        self.is_processing = True
        response = await self.analyze_image(image, ...)
        self.is_processing = False
```

**行为表现**：

| 推理耗时 | 系统行为 | 实际帧率 |
|---------|---------|---------|
| < 1 秒 | 每秒成功推理 | ~1 FPS |
| 1~2 秒 | 部分帧被跳过 | ~0.5 FPS |
| > 2 秒 | 连续跳过，直到完成 | 更低 |

#### 2. 延迟监控与主动丢帧

当系统落后超过阈值时，主动丢帧追赶：

```python
max_latency = 2.0  # 可配置
if frame_latency > max_latency:
    while frame_latency > max_latency:
        frame = await self.track.recv()  # 连续丢弃过期帧
        dropped_count += 1
        # 重新计算新帧延迟
```

#### 3. 帧时间戳精确追踪

系统用 **PTS（Presentation Time Stamp）** 精确计算每帧年龄：
```python
frame_time_offset = (frame.pts - first_frame_pts) * time_base
frame_latency = current_time - (first_frame_time + frame_time_offset)
```

**一句话总结**：宁可漏看几帧，也不让延迟累积——这就是实时系统的核心哲学。

### 🔧 降低 Token 消耗的关键技术

#### AdaCodec 预测性视频编码
- 画面没变化？→ 只花极少 token 表示"和上一帧一样"
- 画面有变化？→ 分配完整 token 预算描述新内容

#### 三级记忆系统

| 层级 | 机制 | 作用 |
|------|------|------|
| **实时帧** | 每秒输入 1 帧 | 捕捉当下瞬间 |
| **中期摘要** | 每 N 帧由 summary model 压缩 | 记住最近几分钟 |
| **长期记忆** | 多个中期摘要进一步压缩 | 记住几小时前的事 |

## 🧠 和多模态的关系

### 1. VL = Vision-Language，本身就是多模态

JoyAI-VL 处理多种模态的融合：

| 模态 | 说明 |
|------|------|
| 👁️ **视觉** | 持续视频流输入（摄像头/直播流） |
| 🗣️ **语言** | 文本/语音输出 |
| 🎤 **音频（可选）** | ASR 语音输入 + TTS 语音输出 |

### 2. 多了一个关键维度：时间

| 特性 | 传统 VL 模型 | JoyAI-VL-Interaction |
|------|-------------|---------------------|
| **交互方式** | 你问我答（回合制） | 自主判断何时开口 |
| **时间维度** | 离线处理 | 实时流式处理 |
| **决策机制** | 每次都被 prompt 触发 | 每秒自主决策 |
| **Token 消耗** | 与视频长度成正比 | AdaCodec 动态分配 |
| **记忆能力** | 受上下文窗口限制 | 三级摘要压缩（无限长度） |

### 3. 交互决策也是多模态的一部分

模型输出不仅是文本，还有**三态决策信号**：
```
视频帧 → [模型] → 三选一：
         ├── 🗣️ Speak（说话）
         ├── 🤫 Silent（沉默）
         └── 🤖 Delegate（委派）
```

### 🎓 通俗类比

**传统多模态模型**：给 AI 发一段视频，然后问"视频里发生了什么？"——AI 看完后回答你。

**JoyAI-VL**：请了一个**保安坐在监控室**——持续盯着屏幕，没事时不说话，看到异常立刻喊你，搞不定就叫更专业的同事。

这就是多模态从"被动问答"到"主动在场"的进化。

## 📚 专业词汇通俗解释

### Speculative Decoding / 推测解码（对比参考）
*注：此为前文 DSpark 的概念，此处仅作对比参考。JoyAI-VL 不使用推测解码，而是使用流式推理。*

### Vision-Language Model (VLM) / 视觉-语言模型
**一句话总结**：能"看懂图片"并"用语言描述"的 AI 模型。
**通俗类比**：像一个既能看画又能解说的讲解员。
**JoyAI 中的实现**：基于 JoyAI-VL-8B，在 Qwen3-VL 基础上训练交互行为。

### Turn-based vs Streaming / 回合制 vs 流式
**一句话总结**：回合制是"你问我答"，流式是"边看边说"。
**JoyAI 创新**：从回合制升级为流式——模型每秒自主决定是否开口。

### WebRTC / 网页实时通信
**一句话总结**：浏览器直接传输音视频流的技术，无需安装插件。
**JoyAI 中的体现**：webui 服务通过 WebRTC 接收摄像头画面，实时推送到推理服务。

### PTS (Presentation Time Stamp) / 显示时间戳
**一句话总结**：视频帧自带的"出生时间"标记。
**JoyAI 中的用途**：精确计算帧的"年龄"，判断是否过期需要丢弃。

### Async Inference / 异步推理
**一句话总结**：推理在后台进行，不阻塞前台接收新数据。
**JoyAI 实现**：`asyncio.create_task(self.vlm_service.process_frame(...))`，视频流持续接收，推理任务异步执行。

### AdaCodec / 自适应视频编码
**一句话总结**：对可预测的帧用极少 token，对有变化的帧用完整 token。
**通俗类比**：视频压缩中的 P 帧/B 帧概念——画面没变就不重复描述。

### Memory Compression / 记忆压缩
**一句话总结**：把大量视觉信息压缩成文字摘要，避免上下文爆炸。
**JoyAI 实现**：三级压缩（实时帧 → Chunk 摘要 → 长期记忆），让 8B 模型能处理无限长视频流。

### Delegation / 任务委派
**一句话总结**：遇到难题时交给更强的模型或工具处理。
**JoyAI 实现**：模型输出 `</delegate> task` 信号，系统转发给 background-agent 异步执行。

### Frame Dropping / 丢帧策略
**一句话总结**：宁可跳过几帧，也不让延迟累积。
**JoyAI 实现**：当推理锁被占用时，新帧直接跳过（`if self._processing_lock.locked(): return`）。

### NanoHermes vs JoyAI 理念对照

| 文章概念 | NanoHermes 实现 | 状态 |
|---------|----------------|------|
| 动态决策 | 工具分发器根据上下文选择工具 | ✅ 类似 |
| 任务委派 | delegate_task 子代理机制 | ✅ 已有 |
| 记忆压缩 | 上下文摘要 + 会话压缩 | ✅ 类似 |
| 实时流式处理 | 不适用（对话式交互） | ⬜ 场景不同 |
