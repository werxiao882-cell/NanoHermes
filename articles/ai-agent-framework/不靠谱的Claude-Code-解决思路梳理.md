# 解决"Claude Code 自审不靠谱"的完整思路梳理

> 基于文章「不靠谱的 Claude Code」+ 评论区 18 条留言，梳理所有解决方案，按成本从低到高排列。

---

## 一、问题本质

**现象**：Claude Code 写完代码后让它自己 review，经常说"没问题"，但实际有 bug（如空数组漏判导致白屏）。

**根因**：同一个模型有"这是我刚写的"认知偏差，带着预设去审，不是在找问题而是在**给自己的结论找证据**。

**结论**：自审有盲区，需要外部视角。

---

## 二、解决方案全景图（7 种，按成本从低到高）

### 方案 1：开新对话让 Claude 自审 🟢 零成本

| 维度 | 说明 |
|------|------|
| **提出者** | zyh、小zh |
| **做法** | 写完代码后，开一个新对话（新 session），把代码贴进去让 Claude review |
| **原理** | 清除上下文中的"我刚刚写了这段代码"的记忆，相当于换一个"失忆的 Claude"来看 |
| **作者实测** | ❌ 效果一般般，不如换模型 |
| **为什么不理想** | 虽然清除了上下文，但底层模型的训练偏好/推理模式没变，还是同一个"脑子" |

**适用场景**：快速小改、不想配置任何东西时先用这个试试。

---

### 方案 2：用 Review Skill 约束 Claude 🟢 低成本

| 维度 | 说明 |
|------|------|
| **提出者** | 吴易易、Mistletoe、余努力 |
| **做法** | 给 Claude 加载一个专门的 code review 技能包（SKILL.md），明确定义 review 检查清单 |
| **具体 Skill** | `superpowers/requesting-code-review`（余努力推荐） |
| **原理** | 通过 prompt 约束，强制 Claude 按照 checklist 逐项检查，减少"轻轻放过"的概率 |
| **效果预期** | 比裸审好，但仍然受限于同一个模型的盲区 |

**适用场景**：不想引入第二个模型，但想让 review 更系统化。

**NanoHermes 实现**：
```
skills/
└── code-review/
    └── SKILL.md    # 定义 review 检查项、边界条件清单
```

---

### 方案 3：Claude 自身 Sub-Agent 并行 🟡 中成本

| 维度 | 说明 |
|------|------|
| **提出者** | 海星海星 |
| **做法** | 利用 Claude 自身的子 Agent 功能和并行 worktree，让一个子 Agent 写、另一个子 Agent 测 |
| **原理** | 同一个模型内部分成两个角色，通过 worktree 隔离工作区 |
| **作者回应** | 自动化已在做，但 Claude 的 quota 经常不够用 |
| **局限性** | 本质还是同一个模型的"分身"，盲区可能重叠 |

**适用场景**：Claude quota 充足、不想引入其他模型时。

---

### 方案 4：异构模型交叉审查（作者方案）🟡 中成本 ⭐ 推荐

| 维度 | 说明 |
|------|------|
| **提出者** | 文章作者 |
| **支持者** | 陈立 Leo（Multi-Agent 就要不同模型，不然伪协作）、江米（差异越大效果越好） |
| **做法** | Claude 写代码，Codex 专门审查，通过共享状态文件接力 |
| **工作流** | ```
Claude 写 → 状态文件设为"待审" → Codex 审 → 写回问题 → 状态切回"待改" → Claude 改 → 循环
``` |
| **原理** | 不同模型体系 → 不同训练数据 → 不同盲区 → 交叉覆盖 |
| **作者实测** | ✅ 最有效，很多 Claude 放过去的问题 Codex 能一眼拎出来 |
| **代价** | Token 消耗增加、时间稍慢 |
| **关键细节** | 谁也不指挥谁，全靠状态机推进；共享一个文件夹 + 约定协议 |

**实现方式**（作者亲述）：
> 就是共享一个文件夹，约定好协议。你把这篇文章分别喂给 CC 和 Codex 就会帮你弄了。

**NanoHermes 实现路径**：
```
# 通过 delegate_task 实现
delegate_task(
    goal="审查 Claude 写的代码",
    toolsets=["terminal", "file"],
    model="openai/gpt-4o"  # 与主模型不同
)
```

---

### 方案 5：Codex 写 + Claude 审（反向交叉）🟡 中成本

| 维度 | 说明 |
|------|------|
| **提出者** | 小张 |
| **做法** | 和方案 4 反过来——Codex 写代码，Claude 来测试/审查 |
| **作者态度** | 作者目前是 Claude 写 + Codex 审，但小张的方案同样有效 |
| **原理** | 异构交叉，方向无所谓，关键是两个不同模型 |

**适用场景**：Codex 写代码能力更强时采用。

---

### 方案 6：多模型轮转审查 🟠 高成本

| 维度 | 说明 |
|------|------|
| **做法** | 3 个或更多模型轮流审查同一份代码（如 Claude → Codex → Gemini） |
| **原理** | 模型越多，盲区覆盖越全面 |
| **代价** | Token 消耗成倍增长，时间大幅增加 |
| **适用场景** | 关键业务代码、安全敏感代码 |

---

### 方案 7：人类 + AI 双重审查 🔴 最高成本

| 维度 | 说明 |
|------|------|
| **做法** | AI 交叉审查后，再由资深工程师最终把关 |
| **原理** | AI 负责初筛，人类负责判断"哪些问题是真正需要改的" |
| **代价** | 人工成本最高 |
| **适用场景** | 生产环境上线前的最终把关 |

## 五、推荐落地路径

**如果你是个人开发者**（低成本优先）：
```
先用方案②（Review Skill）→ 效果不够上方案④（异构交叉）
```

**如果你是团队**（质量优先）：
```
方案④（异构交叉）作为日常 → 方案⑦（人类把关）作为上线前最终关卡
```

**NanoHermes 用户**：
```bash
# 1. 创建 review skill
skills/code-review/SKILL.md

# 2. 配置两个不同模型的 provider
.env 中配置 DASHSCOPE_BASE_URL（Claude）+ OpenAI 兼容端点（Codex/GPT）

# 3. 通过 delegate_task 实现交叉审查
delegate_task(goal="审查代码", model="不同模型")
```

## 六、状态文件协议（方案④实现细节）

作者提到的"共享文件夹 + 约定协议"可以这样实现：

```
workspace/
├── code/              # 代码文件
├── status.json        # 状态机文件
└── review.md          # 审查结果

# status.json 格式
{
  "state": "codex_reviewing",  // idle | claude_writing | codex_reviewing | done
  "last_updated_by": "claude",
  "timestamp": "2026-06-28T22:41:00"
}
```

**流程**：
1. Claude 写完代码 → 把 `status.json` 的 `state` 改为 `codex_reviewing`
2. Codex 监控到状态变化 → 读取代码 → 写入 `review.md`
3. Codex 把 `state` 改为 `claude_writing`（如果有问题）或 `done`
4. Claude 根据 `review.md` 修改代码 → 循环
