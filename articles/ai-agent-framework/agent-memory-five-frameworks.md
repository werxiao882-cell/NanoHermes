# 一篇讲清：Agent Memory 五种主流框架附开源

> 来源：AI大模型蛋蛋（微信公众号）
> 整理时间：2026-06-27

## 为什么 Agent 需要 Memory？

想象一下，你每次跟同事说话，他都会问你"你是谁？我们上次聊了什么？"——这种同事你肯定想拉黑。但早期的 AI Agent 就是这样：**无状态、无记忆、每次对话从零开始**。

Agent Memory（智能体记忆）就是为了解决这个问题而生的。它让 AI 从"一次性问答工具"进化为"能记住你的数字员工"。

---

## 🧠 Agent Memory 的三种记忆类型

| 记忆类型 | 类比 | 技术实现 | 特点 |
|---------|------|---------|------|
| **短期记忆** | 工作记忆（RAM） | Context Window（上下文窗口） | 容量有限，对话结束即消失 |
| **长期记忆** | 硬盘存储 | 向量数据库 + 检索 | 持久化，可跨会话检索 |
| **程序性记忆** | 肌肉记忆 | 技能/工具调用记录 | 知道"怎么做"而非"是什么" |

---

## 📦 五种主流 Agent Memory 框架

### 1️⃣ Mem0 — 通用记忆层

**GitHub**: https://github.com/mem0ai/mem0

**一句话介绍**：Mem0 是一个开源的"记忆中间件"，为任何 AI Agent 添加长期记忆能力，就像给大脑装了个硬盘。

**核心特性**：
- **三层记忆架构**：短期记忆 → 长期记忆 → 程序性记忆
- **自适应更新**：自动决定哪些信息值得记住，哪些该遗忘
- **多用户支持**：每个用户有独立的记忆空间
- **即插即用**：支持 OpenAI、Anthropic、Ollama 等主流模型

**工作原理**：
```
用户输入 → Mem0 提取关键信息 → 向量存储 → 下次对话时检索相关记忆 → 增强 Agent 回复
```

**适用场景**：个人 AI 助手、客服系统、教育应用

---

### 2️⃣ MemGPT / Letta — 虚拟上下文管理

**GitHub**: https://github.com/cpacker/MemGPT（现更名为 Letta）

**一句话介绍**：让 LLM 拥有"无限上下文"，通过自动管理内存来突破 token 限制。

**核心创新**：
- **分层记忆系统**：模拟操作系统的内存管理（RAM ↔ 硬盘）
- **自我编辑记忆**：Agent 能主动修改自己的记忆
- **对话历史压缩**：自动总结旧对话，释放上下文空间
- **多文档管理**：同时管理多个对话线程的记忆

**工作原理**：
```
主上下文（活跃记忆） ←→ 外部存储（归档记忆）
        ↑ 自动交换 ↑
    LLM 控制上下文窗口
```

**适用场景**：长对话、多轮任务、复杂项目管理

---

### 3️⃣ LangGraph Memory — 工作流记忆

**GitHub**: https://github.com/langchain-ai/langgraph

**一句话介绍**：LangChain 生态中的记忆模块，专为多步骤工作流设计。

**核心特性**：
- **图结构记忆**：用图数据库存储 Agent 的决策路径
- **检查点机制**：可随时回退到之前的状态
- **持久化存储**：支持 SQLite、PostgreSQL 等后端
- **与 LangChain 深度集成**：无缝衔接现有 Agent 开发流程

**工作原理**：
```
Agent 执行步骤 → 每个步骤状态保存为检查点 → 可回溯/重试/分支
```

**适用场景**：复杂工作流、需要状态回滚的任务、多步骤推理

---

### 4️⃣ Zep — 长期记忆即服务

**GitHub**: https://github.com/getzep/zep

**一句话介绍**：专为 AI Agent 设计的长期记忆和知识图谱服务。

**核心特性**：
- **自动记忆提取**：从对话中自动提取实体、关系、事实
- **知识图谱**：构建用户画像和知识网络
- **多模态支持**：支持文本、向量、图数据混合存储
- **API 优先**：REST API，可集成任何 Agent 框架

**工作原理**：
```
对话输入 → NER 实体识别 → 关系抽取 → 知识图谱更新 → 检索增强回复
```

**适用场景**：CRM 系统、个性化推荐、用户画像构建

---

### 5️⃣ Eliza — 社交 Agent 记忆框架

**GitHub**: https://github.com/elizaOS/eliza

**一句话介绍**：Twitter/X 上爆火的社交 Agent 框架，内置记忆和人格系统。

**核心特性**：
- **人格记忆**：Agent 有固定的人格特征和记忆偏好
- **社交记忆**：记住与不同用户的互动历史
- **多平台支持**：Twitter、Discord、Telegram 等
- **社区驱动**：开源社区活跃，插件生态丰富

**工作原理**：
```
社交互动 → 提取用户特征 → 更新人格记忆 → 生成个性化回复
```

**适用场景**：社交机器人、虚拟网红、社区运营

---

## 📊 五种框架对比

| 维度 | Mem0 | Letta (MemGPT) | LangGraph Memory | Zep | Eliza |
|------|------|---------------|-----------------|-----|-------|
| **核心定位** | 通用记忆层 | 无限上下文 | 工作流状态 | 知识图谱 | 社交记忆 |
| **记忆类型** | 三层架构 | 分层管理 | 检查点 | 实体关系 | 人格+社交 |
| **开源协议** | MIT | MIT | MIT | MIT | MIT |
| **部署难度** | ⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| **适合场景** | 通用 | 长对话 | 工作流 | 知识管理 | 社交 |
| **存储后端** | 向量DB | 本地/云端 | SQL/文件 | 图数据库 | 本地/云端 |

---

## 🔧 如何选择？

- **想要快速给现有 Agent 加记忆** → 选 **Mem0**（最简单）
- **需要超长对话** → 选 **Letta**（虚拟上下文）
- **复杂多步骤任务** → 选 **LangGraph Memory**（状态管理）
- **需要知识图谱** → 选 **Zep**（实体关系提取）
- **社交/对话机器人** → 选 **Eliza**（人格记忆）

## 🚀 快速上手 Mem0 示例

```python
from mem0 import Memory

# 初始化记忆
m = Memory()

# 添加记忆
m.add("用户叫张三，喜欢 Python 编程")
m.add("张三的项目是用 FastAPI 做后端")

# 检索相关记忆
results = m.search("张三的项目是什么？")
print(results)
# 返回：张三的项目是用 FastAPI 做后端
```

## 📚 专业词汇通俗解释（结合 NanoHermes 项目源码）

### 1. Agent Memory（智能体记忆）

**一句话**：就是 AI 的"笔记本"，让它能记住之前聊过什么。

**NanoHermes 中的实现**：NanoHermes 通过 `memory` 工具实现类似功能，将重要信息保存到持久化存储（`~/.hermes/memories/`），每次对话自动加载。相当于 Agent 有个随身携带的小本本，重要事情都记下来。

**类比**：就像你每次开会前翻看之前的会议记录，而不是每次开会都问"我们上次说了啥？"

### 2. Context Window（上下文窗口）

**一句话**：AI 每次能"同时看到"的文本量上限。

**类比**：就像人的工作记忆，一次只能记住 7±2 个信息块。如果对话太长，最早的内容就会被"挤出"窗口。

**NanoHermes 中的体现**：NanoHermes 通过 `session_search` 工具在需要时查找历史对话，而不是把所有历史都塞进上下文窗口——相当于"记不住的就去翻档案"。

### 3. Vector Database（向量数据库）

**一句话**：把文字变成数字坐标，按"相似度"快速查找相关内容。

**类比**：想象图书馆不是按书名排序，而是按"内容相似度"排列。你找一本关于"猫"的书，系统会自动把"狗""宠物""动物"相关的书也找出来。

### 4. Knowledge Graph（知识图谱）

**一句话**：用"节点+连线"的方式存储实体之间的关系。

**类比**：就像家谱图，记录谁是谁的亲戚。知识图谱记录"张三是 Python 开发者"、"Python 是编程语言"、"FastAPI 是 Python 框架"这样的关系网。

**NanoHermes 中的体现**：NanoHermes 的 `memory` 和 `skill` 系统虽然不用图数据库，但通过 YAML 文件存储结构化信息（如技能定义、记忆条目），实现了类似的效果。

### 5. Checkpoint（检查点）

**一句话**：保存当前状态，随时可以回退到这个时间点。

**类比**：就像玩游戏的存档功能。打 Boss 前存个档，失败了读档重来。

**NanoHermes 中的体现**：NanoHermes 通过 `todo` 工具管理任务状态，每个 todo 项有明确的 status（pending/in_progress/completed），相当于轻量级检查点。

### 6. NER（命名实体识别）

**一句话**：从文本中自动提取人名、地名、机构名等关键信息。

**类比**：就像你用荧光笔在书上划重点，NER 就是 AI 版的荧光笔，自动标出"张三"（人名）、"北京"（地名）、"微软"（机构名）。

## 🔄 NanoHermes vs 文章理念的对照

| 文章概念 | NanoHermes 实现 | 状态 |
|---------|----------------|------|
| 持久化记忆 | `memory` 工具（SQLite 存储） | ✅ 已实现 |
| 技能/工具记忆 | `skill` 系统（SKILL.md） | ✅ 已实现 |
| 对话历史检索 | `session_search` 工具 | ✅ 已实现 |
| 上下文窗口管理 | 隐式（通过工具调用控制） | ✅ 已实现 |
| 知识图谱 | 无（使用结构化 YAML） | ⬜ 可优化 |
| 自动记忆提取 | 手动保存（需 agent 主动调用） | ⬜ 可自动化 |
| 多用户记忆隔离 | 无（单用户设计） | ⬜ 暂不需要 |

## 💡 可以借鉴文章改进的方向

1. **自动记忆提取**：当前 NanoHermes 需要 agent 主动调用 `memory` 工具保存信息，可以借鉴 Mem0 的思路，在对话中自动识别值得保存的信息。
2. **记忆优先级**：参考 Mem0 的三层记忆架构，区分"重要事实"（如用户偏好）和"临时信息"（如某次对话的细节）。
3. **记忆过期机制**：长期记忆会越来越多，需要设计"遗忘"策略，避免检索时信息过载。
