# Transformer 显存占用深度分析

> 为什么 Transformer 模型如此"吃"显存？
> 分析时间: 2026年6月

---

## 一、一句话回答

**Transformer 的显存占用 = 模型参数 + KV Cache + 激活值 + 梯度 + 优化器状态**

其中 **KV Cache 随序列长度线性增长**，**Attention 激活值随序列长度平方增长**——这是 Transformer 显存占用的两个核心瓶颈。

---

## 二、显存占用的五大组成部分

### 2.1 模型参数显存（权重存储）

**这是什么？** 模型训练好后保存下来的"知识"，就是那些权重矩阵。

**显存计算：**
```
参数量 × 精度字节数 = 显存占用

fp32:  4 bytes/param
bf16:  2 bytes/param (目前主流)
fp16:  2 bytes/param
int8:  1 byte/param  (量化)
int4:  0.5 bytes/param (量化)
```

**以 GPT-3 175B 为例：**

| 精度 | 显存占用 |
|------|---------|
| fp32 | 700 GB |
| bf16 | 350 GB |
| int8 | 175 GB |
| int4 | 87.5 GB |

**逐层拆解（GPT-3 175B，96层，hidden=12288）：**

| 组件 | 参数量 | 显存 (bf16) |
|------|--------|------------|
| Embedding | 617.6M | 1.24 GB |
| 每层 Transformer | 1.81B | 3.62 GB |
| 96层总计 | 174B | 348 GB |

**每层 Transformer 的参数分布：**

| 子模块 | 参数量 | 占比 |
|--------|--------|------|
| Q/K/V/O 线性层 | 4 × (d² + d) | 33% |
| FFN 两个线性层 | 2 × (d×d_ff + d_ff) | 66% |
| LayerNorm × 2 | 4d | <1% |

**关键洞察：FFN 占了 2/3 的参数！** 这就是为什么缩小 FFN 维度能显著减少显存。

### 2.2 KV Cache 显存（注意力机制的核心瓶颈）

**这是什么？** 生成文本时，模型需要记住之前所有 token 的 Key 和 Value 向量，这就是 KV Cache。

**为什么需要它？** 
```
生成第 t 个 token 时:
  需要计算 token_t 与 token_1...token_{t-1} 的注意力
  如果不缓存，每次都要重新计算前 t-1 个 token 的 K 和 V → 巨慢！
  缓存后，只需计算 token_t 的 Q，然后与缓存的 K/V 做注意力 → 快！
```

**显存计算：**
```
KV Cache 大小 = batch_size × seq_len × n_layers × 2 × n_heads × d_head × 2

解释:
  batch_size: 批次大小
  seq_len: 序列长度
  n_layers: Transformer 层数
  2: K 和 V 两个矩阵
  n_heads: 注意力头数
  d_head: 每个头的维度 (d_model / n_heads)
  2: bf16 精度 (2 bytes)
```

**GPT-3 175B 的 KV Cache（batch=1）：**

| 序列长度 | KV Cache 显存 |
|---------|--------------|
| 256 | 1.15 GB |
| 1,024 | 4.61 GB |
| **4,096** | **18.00 GB** |
| 8,192 | 36.86 GB |
| 16,384 | 73.73 GB |
| 32,768 | 147.46 GB |

**关键洞察：**
- KV Cache **随序列长度线性增长**
- 长序列时，KV Cache 可能超过模型参数本身！
- batch_size 增大时，KV Cache 也线性增长

**类比：** 
- 模型参数是"字典"（固定大小）
- KV Cache 是"草稿纸"（写得越长，纸用得越多）
- 生成 32K token 需要的草稿纸，比字典还大 4 倍！

### 2.3 激活值显存（前向传播中间结果）

**这是什么？** 模型计算过程中产生的临时数据，比如每层的输入、输出、注意力分数等。

**训练时 vs 推理时：**
- **推理时：** 每层计算完就可以释放，显存占用很小（~0.1 GB）
- **训练时：** 需要保存所有中间结果用于反向传播，显存占用巨大！

**激活值的主要组成部分：**

| 组件 | 显存公式 | GPT-3 175B (seq=4096) |
|------|---------|----------------------|
| 线性层输入输出 | 4 × bs × seq × d | 384 MB/层 |
| FFN 输入输出 | 2 × bs × seq × d_ff | 768 MB/层 |
| LayerNorm 输入输出 | 2 × bs × seq × d | 96 MB/层 |
| **Attention scores** | **bs × seq × seq** | **32 MB/层** |
| 每层总计 | - | ~1.38 GB/层 |
| **96层总计** | - | **~129 GB** |

**Attention scores 的 O(seq²) 问题：**

```
Attention 需要计算所有 token 两两之间的注意力分数:
  seq=4096: 4096 × 4096 = 16.8M 个分数
  seq=8192: 8192 × 8192 = 67.1M 个分数 (4倍!)
  seq=32768: 32768 × 32768 = 1.07B 个分数 (64倍!)
```

**Flash Attention 的拯救：**

```
原始 Attention: O(seq²) 显存 (需要存储完整的注意力矩阵)
Flash Attention: O(seq) 显存 (分块计算，不存储完整矩阵)

seq=4096 时:
  原始: 0.03 GB
  Flash: 0.0078 GB (节省 75%)
```

### 2.4 梯度显存（训练时）

**这是什么？** 反向传播时计算的梯度，用于更新模型参数。

**显存计算：**
```
梯度显存 = 参数量 × 4 bytes (fp32)

GPT-3 175B: 175B × 4 = 700 GB
```

**为什么梯度要用 fp32？**
- 精度太低会导致梯度消失/爆炸
- fp16 的动态范围不够，容易溢出
- bf16 比 fp16 好一些，但大多数框架仍用 fp32 存梯度

### 2.5 优化器状态（训练时，最吃显存的部分！）

**这是什么？** AdamW 优化器需要维护的额外状态。

**AdamW 的显存占用（每参数）：**

| 组件 | 精度 | 字节数 |
|------|------|--------|
| 参数 (fp16) | fp16 | 2 bytes |
| 梯度 | fp32 | 4 bytes |
| Momentum (一阶矩) | fp32 | 4 bytes |
| Variance (二阶矩) | fp32 | 4 bytes |
| 主参数副本 | fp32 | 4 bytes |
| **总计** | - | **18 bytes/param** |

**GPT-3 175B 的优化器状态：**

| 组件 | 显存 |
|------|------|
| 参数 (fp16) | 349 GB |
| 梯度 (fp32) | 698 GB |
| Momentum (fp32) | 698 GB |
| Variance (fp32) | 698 GB |
| 主参数 (fp32) | 698 GB |
| **总计** | **3,142 GB** |

**关键洞察：优化器状态是模型参数的 9 倍！**

## 四、显存优化技术

### 4.1 模型并行（解决参数显存）

**问题：** 单个 GPU 装不下模型参数。

**方案：**
```
张量并行 (TP): 把大矩阵切分到多个 GPU
  Q = [Q1, Q2, Q3, Q4] → 4 个 GPU 各算一部分
  效果: 显存 / GPU数

流水线并行 (PP): 把模型层切分到多个 GPU
  GPU0: Layer 0-23
  GPU1: Layer 24-47
  GPU2: Layer 48-71
  GPU3: Layer 72-95
  效果: 显存 / GPU数
```

### 4.2 ZeRO（解决优化器状态显存）

**DeepSpeed ZeRO 的三阶段优化：**

| 阶段 | 分片内容 | 效果 |
|------|---------|------|
| ZeRO-1 | 优化器状态 | 优化器显存 / GPU数 |
| ZeRO-2 | 优化器 + 梯度 | (优化器+梯度) / GPU数 |
| ZeRO-3 | 优化器 + 梯度 + 参数 | 全部显存 / GPU数 |

**GPT-3 175B + ZeRO-3 (8卡)：**
```
单卡显存: 4,337 GB / 8 = 542 GB/GPU
仍需要 8×A100 80GB + CPU Offload
```

### 4.3 KV Cache 优化

| 技术 | 原理 | 效果 |
|------|------|------|
| PagedAttention | 类似操作系统虚拟内存，按需分配 | 减少碎片，提升吞吐 |
| KV Cache 量化 | int8/int4 量化 K 和 V | 显存减半/四分之一 |
| Prefix Caching | 共享相同前缀的请求共用 KV Cache | 多请求时显著节省 |
| Sliding Window | 只保留最近 N 个 token 的 KV | 长序列时显存 O(N) 而非 O(seq) |

### 4.4 激活值优化

| 技术 | 原理 | 效果 |
|------|------|------|
| 梯度检查点 (Activation Checkpointing) | 只保存部分激活值，其余重新计算 | 显存减少 50-80%，计算增加 20-30% |
| Flash Attention | 分块计算，不存储完整 attention 矩阵 | O(seq²) → O(seq) |
| 混合精度训练 | 前向用 fp16/bf16，反向用 fp32 | 激活值显存减半 |

### 4.5 量化（推理时最有效）

| 量化方式 | 精度 | 显存占比 | 质量损失 |
|---------|------|---------|---------|
| fp16/bf16 | 16 bit | 100% | 0% |
| int8 | 8 bit | 50% | <1% |
| int4 | 4 bit | 25% | 1-3% |
| int4 + AWQ | 4 bit (智能量化) | 25% | <1% |

**Llama-2 70B 量化效果：**
```
fp16:  140 GB → 需要 2×A100 80GB
int8:  70 GB  → 需要 1×A100 80GB
int4:  35 GB  → 需要 1×RTX 4090 24GB + Offload
```

## 五、通俗类比

### 5.1 Transformer 显存 = 厨房做菜

| 组件 | 厨房类比 | 显存特点 |
|------|---------|---------|
| 模型参数 | 菜谱（固定不变） | 固定大小，加载一次 |
| KV Cache | 切好的菜（越做越多） | 随序列长度线性增长 |
| 激活值 | 锅碗瓢盆（用完就洗） | 推理时小，训练时大 |
| 梯度 | 尝味记录（训练需要） | 只有训练时有 |
| 优化器状态 | 调料储备（Adam 需要） | 训练时最大，是参数的 9 倍 |

### 5.2 为什么长序列这么吃显存？

**类比：** 
- 短序列 = 写日记，每天写一行，本子很小
- 长序列 = 写历史书，每天写一章，需要整个图书馆

**数学上：**
```
seq=4K:  KV Cache = 18 GB (一本厚书)
seq=32K: KV Cache = 147 GB (一个书架)
seq=128K: KV Cache = 589 GB (整个图书馆!)
```

## 六、实际场景分析

### 6.1 本地运行 7B 模型需要什么配置？

**Llama-2 7B (seq=4096)：**

| 组件 | 显存 |
|------|------|
| 模型参数 (bf16) | 14 GB |
| KV Cache (batch=1) | 2 GB |
| 激活值 | 0.1 GB |
| **总计** | **16.1 GB** |

**结论：** RTX 4090 (24GB) 可以流畅运行，还有余量给更大 batch 或更长序列。

### 6.2 本地运行 70B 模型需要什么配置？

**Llama-2 70B (seq=4096)：**

| 组件 | 显存 |
|------|------|
| 模型参数 (bf16) | 140 GB |
| KV Cache (batch=1) | 10 GB |
| 激活值 | 0.1 GB |
| **总计** | **150 GB** |

**方案选择：**

| 方案 | 硬件需求 | 成本 |
|------|---------|------|
| fp16 双卡 | 2×A100 80GB | ~$30,000 |
| int8 单卡 | 1×A100 80GB | ~$15,000 |
| int4 单卡 | 1×RTX 4090 24GB (需 Offload) | ~$2,000 |
| int4 + CPU Offload | RTX 4090 + 64GB 内存 | ~$2,500 |

### 6.3 训练 7B 模型需要什么配置？

**Llama-2 7B 训练 (batch=1, seq=4096)：**

| 组件 | 显存 |
|------|------|
| 模型参数 (fp16) | 14 GB |
| 梯度 (fp32) | 28 GB |
| 优化器状态 (AdamW) | 56 GB |
| 激活值 | ~10 GB |
| KV Cache | 2 GB |
| **总计** | **~110 GB** |

**方案：**
```
ZeRO-3 + 梯度检查点 + 混合精度:
  4×A100 80GB: 每卡 ~27.5 GB ✅ 可行
  2×A100 80GB: 每卡 ~55 GB ✅ 可行 (紧凑)
  1×A100 80GB: 80 GB < 110 GB ❌ 不够
```

## 七、总结：为什么 Transformer 这么吃显存？

| 原因 | 本质 | 影响 |
|------|------|------|
| **参数多** | 模型越大能力越强 | 7B→14GB，70B→140GB |
| **KV Cache** | 生成时需要记住所有历史 | 随序列长度线性增长 |
| **Attention O(seq²)** | 每个 token 要和所有 token 算注意力 | 序列翻倍，显存×4 |
| **训练要梯度+优化器** | 反向传播需要额外 9 倍显存 | 训练显存是推理的 12 倍 |
| **FFN 占 2/3 参数** | 前馈网络比注意力更吃参数 | 缩小 FFN 最有效 |

**一句话总结：**
> Transformer 的显存占用是由其架构本质决定的——**注意力机制需要看所有 token**（KV Cache 线性增长），**训练需要反向传播**（梯度+优化器 9 倍参数），**模型越大效果越好**（参数与显存成正比）。这不是 bug，是 feature。
