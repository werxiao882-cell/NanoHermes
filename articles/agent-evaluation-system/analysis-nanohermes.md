# Agent 评测体系 vs NanoHermes 项目分析报告

> 基于微信文章《Agent 评测需要体系化》（2026 年第 28 篇）的深度对比分析

---

## 一、文章核心观点总结

文章提出 **Agent 评测必须体系化**，核心论点是：Agent 从 Demo 到生产可用，要跨过三道门槛：

1. **非确定性**：同样输入不一定同样输出
2. **黑盒化**：内部决策过程不透明
3. **错误级联放大**：前一步小错会在后续被放大

文章构建了一套完整的 **9 模块评测平台架构**，涵盖：数据集管理 → 用例执行 → Trace 采集 → 评分 → Badcase 分析 → 根因定位 → 优化建议 → 工单流转 → 线上灰度。

## 三、NanoHermes 最值得学习的 8 个方向

### 🔥 第一优先级：Agent 行为评测系统

**为什么重要：** 现有 97 个单元测试保证了代码正确性，但 **不能保证 Agent 行为质量**。换个模型、改个 Prompt、更新个 Skill，Agent 可能产生完全不同的行为，但单元测试不会报错。

**文章建议的做法：**
- 建立 Golden Set（基线评测集）：50-200 条核心用例
- 每次变更（模型/Prompt/Skill/工具参数）后自动回归
- 关注 **连续成功率**（每次都成功）而非仅 **至少一次成功率**

**NanoHermes 怎么落地：**
```
tests/evaluation/
├── golden_set/              # 基线用例
│   ├── tool_execution.yaml   # 工具调用正确性
│   ├── skill_trigger.yaml    # Skill 触发准确性
│   ├── conversation.yaml     # 多轮对话质量
│   └── safety.yaml           # 安全护栏
├── runners/                  # 评测执行器
│   ├── behavior_runner.py    # 行为一致性
│   └── regression_runner.py  # 版本回归
└── scorers/                  # 评分器
    ├── rule_scorer.py        # 规则评分（确定性）
    └── judge_scorer.py       # LLM-as-Judge（语义）
```

### 🔥 第二优先级：Skill 专项评测

**为什么重要：** NanoHermes 已经有很好的 Skill 系统，但 **没有评测 Skill 质量的机制**。Skill 的 SKILL.md 内容质量、触发准确性、执行效果都无法量化。

**文章中的 Skill 检查项：**
- **触发准确率**：该触发时触发了吗？不该触发时误触了吗？
- **流程正确率**：触发后是否走了正确的流程？
- **参数正确率**：工具调用参数是否正确？
- **产物质量**：最终产出是否可用？
- **异常容错**：异常时是否能降级？

**NanoHermes 怎么落地：**
在 `skills/` 每个 Skill 的 SKILL.md 中加入 `evaluation` 章节：
```yaml
evaluation:
  trigger_test_cases:    # 触发测试
    - should_trigger: ["帮我写个 Python 脚本"]
    - should_not_trigger: ["今天天气怎么样"]
  execution_test_cases:  # 执行测试
    - input: "分析这个 CSV"
      expected_tools: ["read_file", "execute_code"]
      expected_output_contains: ["分析结果"]
```

### 🔥 第三优先级：Trace 结构化记录

**为什么重要：** 文章强调 **"把 Trace 输出当作被测 Agent 的标准能力"**。NanoHermes 目前 JSONL 存储的是对话消息，但缺少结构化的执行轨迹记录。

**建议记录内容：**
- 工具调用：名称、入参、返回值、耗时、错误
- 计划步骤：Agent 的决策链
- 技能加载：加载了哪些 Skill、为什么
- 上下文压缩：何时触发、压缩了多少
- 错误恢复：遇到错误后的处理路径

### 第四优先级：Badcase 根因分析 Pipeline

**文章的五步链路：**
1. **证据汇总**：收集 sessionId、Trace、日志
2. **范围收敛**：问题现象 × 功能模块映射表
3. **分模块诊断**：逐个检查 input/output/prompt/工具返回
4. **责任判定**：主责、次责、修复建议
5. **结构化落盘**：写入任务记录，可查询可统计

**NanoHermes 可以借鉴：**
- 利用现有的 `error_classifier.py` 作为第一步
- 扩展为 Agent 行为级根因分析（不仅是 API 错误）
- 与 insights engine 结合，生成评测报告

### 第五优先级：LLM-as-Judge 评分

**为什么需要：** 规则评分（如工具是否调用、参数是否正确）只能覆盖确定性维度。语义质量、回复相关性、策略合规性等需要 LLM 评判。

**三层评分策略：**
1. **规则评分**（优先）：确定性检查，零成本
2. **LLM Judge**：语义和策略维度，中等成本
3. **人工评分**：高风险/低置信样本路由

### 第六优先级：反馈生产线

**文章的核心观点：** "发现问题本身不会自动带来收益，只有把问题沉淀成可复用资产，下一轮迭代成本才会下降。"

**三条生产线：**
1. **用例生产线**：Badcase → 回归测试用例
2. **知识生产线**：Badcase → 知识库修订 / Skill 更新
3. **训练生产线**：Badcase → 训练数据 / Prompt 优化

**NanoHermes 天然优势：** 已有 Skill 自进化机制（Curator），可以直接对接反馈生产线。

### 第七优先级：指标体系分级

**P0 指标（上线门禁）：**
- 核心场景通过率 ≥ 95%
- P0 风险数 = 0
- 关键工具参数正确率 ≥ 99%

**P1 指标（版本比较）：**
- 整体质量得分
- 各维度得分对比
- 统计显著性检验

**P2 指标（体验改善）：**
- 回复质量
- 用户满意度
- 长尾场景覆盖

### 第八优先级：持续迭代闭环

**文章的闭环：**
```
评测 → 发现Badcase → 根因定位 → 优化建议 → 修复 → 回归验证 → 线上灰度 → 反馈回流 → 更新评测集
```

NanoHermes 已具备的部分环节可以串联起来：
- `cronjob` 工具可以定时触发评测
- `insights engine` 可以生成报告
- `skills curator` 可以自动更新 Skill
- `session_search` 可以检索历史问题

## 五、专业词汇通俗解释

### Agent 评测相关

**Golden Set（黄金评测集）**
> 一组精心设计的核心测试用例，长期复用，用于版本对比和发布门禁。类比：驾照考试的固定科目，每次考试都考这些，才能公平比较不同人的水平。

**LLM-as-Judge**
> 让大语言模型充当评分员，评判 Agent 输出的质量。类比：请一位经验丰富的老师来批改作文，而不是用自动评分机。

**连续成功率 vs 至少一次成功率**
> 同一任务跑 N 次，每次都成功叫连续成功率；只要有一次成功叫至少一次成功率。生产系统关心连续成功率——用户不会接受"多试几次总有一次成功"。类比：医生做手术，10 次都要成功，不是 10 次中只要有 1 次成功就行。

**Trace（执行轨迹）**
> 记录 Agent 执行过程中的每一步：做了什么决定、调用了什么工具、得到了什么结果。类比：飞机的黑匣子，记录飞行过程中的所有操作和数据。

**Badcase RCA（根因分析）**
> 不仅知道"出错了"，还要知道"为什么错"、"哪个模块的错"、"怎么修"。类比：车坏了，不仅知道"打不着火"，还要查出是电池没电、起动机坏了、还是油路堵塞。

**统计显著性检验**
> 判断两个版本之间的差异是"真实提升"还是"随机波动"。类比：小明这次考试比上次高 3 分，是进步了还是运气好？需要看更多次考试成绩才能判断。

**反馈生产线**
> 把线上失败案例自动转化为可复用的研发资产（测试用例、知识更新、Prompt 优化）。类比：工厂的废品回收线，把废品变成原材料重新利用。
